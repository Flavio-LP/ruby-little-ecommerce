# Story 5.2: Checkout Flow Creates an Order

## Status
Done

## Story
**As a** customer,
**I want** to convert my cart into a finalized order,
**so that** my purchase is recorded and the seller is notified.

## Acceptance Criteria
1. A `Checkout::CreateOrder` Interactor: validates the cart is non-empty, creates `Order` + `LineItem`s from current `CartItem`s within a transaction, clears the cart on success, sets `status: pending`.
2. On success, the customer is redirected to an order confirmation page showing the order summary.
3. A `OrderConfirmationJob` (Sidekiq) is enqueued to send a confirmation notification (email delivery itself can be a stub/log-based `ActionMailer` preview in MVP).
4. A feature spec covers the full cart-to-confirmation flow, including verifying the job was enqueued (`ActiveJob::TestHelper`).
5. Checkout with an empty cart is rejected with a clear error message, not a 500.

## Dev Notes
- Depends on Story 5.1 (`Order`/`LineItem`) and Epic 4 (`Cart`/`CartItem`).
- Reuses the `Interactor` + transaction pattern already established in `Shops::Register` (Story 2.2) — follow the same shape (`context.fail!` on invalid state, wrapped in `ActiveRecord::Base.transaction`).
- No real payment gateway integration in this MVP — `status: pending` is the terminal state for this story; moving to `paid`/`fulfilled` is manual/future work, not in scope here.

### Completion Notes List
- `Checkout::CreateOrder` segue o mesmo padrão de `Shops::Register`/`Carts::AddItem`: transação envolvendo `Order.create!` + `LineItem.create!` por item do carrinho + `cart.cart_items.destroy_all`, com `context.fail!` para carrinho vazio (rollback automático via exceção dentro da transação).
- `OrderConfirmationJob` usa `Order.unscoped.find` deliberadamente — jobs em background não têm `ActsAsTenant.current_tenant` setado (não há request), então o default_scope do tenant bloquearia a busca; já temos o ID específico, então o escape hatch é seguro e documentado no código.
- `OrderMailer` é preview/log-only (sem provedor SMTP real), conforme MVP — `deliver_now` dentro do próprio job, sem enfileirar um segundo job de mailer.
