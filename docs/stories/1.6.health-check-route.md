# Story 1.6: Health-Check Route

## Status
Done

## Story
**As an** operator,
**I want** a `/up` route that verifies the app, database, and Redis connections,
**so that** I have a deployable, observable signal that the foundation actually works end-to-end.

## Acceptance Criteria
1. `GET /up` returns HTTP 200 with a simple body when app, Postgres, and Redis are all healthy.
2. The route returns a non-200 status if the database connection fails.
3. The route requires no authentication.
4. A request spec covers both the healthy and unhealthy paths.

## Dev Notes
- Rails 8 already ships a default `/up` route (`Rails::HealthController`) checking only DB migrations status — this story extends/replaces it to also check Redis connectivity, per FR10 in `docs/prd/requirements.md`.
- Depends on Stories 1.1 (db/redis services) and 1.5 (Redis/Sidekiq wiring) for the Redis connection check to be meaningful.

### Completion Notes List
- `HealthController` herda de `ActionController::Base` diretamente (não de `ApplicationController`), igual ao padrão do `Rails::HealthController` padrão — evita que `allow_browser versions: :modern` bloqueie load balancers/uptime monitors que não enviam um User-Agent de navegador.
- Rota `/up` trocada de `rails/health#show` para `health#show`.
- **Lição operacional registrada para as próximas stories:** initializers/config nova (ex.: `config/initializers/sidekiq.rb`, `devise.rb`) só são carregados no boot — alterações neles exigem `docker compose restart web worker`, não bastam hot-reload. Isso causou um 500 transitório (`undefined method 'devise' for User:Class`) até o restart.
