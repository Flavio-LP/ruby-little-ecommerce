# Story 1.3: Devise Authentication Setup

## Status
Done

## Story
**As a** developer,
**I want** Devise installed with a single `User` model that includes a `role` enum (`seller`/`customer`),
**so that** both sellers and customers can authenticate through the same mechanism while remaining distinguishable for authorization.

## Acceptance Criteria
1. `rails generate devise:install` and `rails generate devise User` have been run.
2. `User` model has a `role` column (integer enum: `seller`, `customer`) with a non-null default and validation enforcing one of the two values.
3. Devise views (registration, sessions) are generated and themed minimally with Bootstrap.
4. A model spec verifies that a `User` cannot be saved without a valid `role`.

## Dev Notes
- Schema detail: `docs/architecture/schema-design.md` — section "User (Devise + tenant/role extensions)". Exact migration: `add_column :users, :role, :integer, null: false, default: 1` (0=seller, 1=customer), `add_column :users, :shop_id, :bigint` + index + FK to `shops` (the `shops` table itself is created in Epic 2 Story 2.1 — this story should add the `shop_id` column as nullable now so the migration order doesn't block, with the FK constraint added once `shops` exists, OR defer the FK addition to Story 2.1 if `shops` table doesn't exist yet. Recommended: add `shop_id` as a plain nullable bigint column with index in this story; add the `add_foreign_key :users, :shops` in Story 2.1 once the `shops` table exists).
- Critical rule: `docs/framework/coding-standards.md` — authorization must go through `Ability`, never inline `role` checks in controllers/views (this story only sets up the data; `Ability` itself is Story 1.4).

## QA Results
**Verdict:** PASS
- `docker compose exec web bin/rails db:migrate` aplicou a migração com sucesso.
- `bundle exec rspec spec/models/user_spec.rb` → 3 examples, 0 failures.
- **Bug ambiental encontrado e corrigido durante esta story (afeta toda a suíte, não só esta):** `docker-compose.yml` define `RAILS_ENV: development` no serviço `web`; como `docker compose exec` herda essa variável, `bundle exec rspec` estava silenciosamente rodando em modo `development` (com a allowlist de hosts restritiva do Rails), causando 403 em qualquer request spec. Corrigido tornando `ENV['RAILS_ENV'] = 'test'` incondicional em `spec/rails_helper.rb` (em vez de `||=`). Sem esse fix, toda story futura com request/feature specs teria o mesmo problema.
